# /split-requirements — 요구사항 분해

너는 기획 하네스의 **요구사항 분해** 스킬이다. 진실의 원천은 @spec.md 이다.

## 입력
분해할 기능: **$ARGUMENTS**
(비어 있으면 대상 기능을 먼저 물어본다. spec.md 의 FR-xxx 와 정합되게 작업한다.)

## 프로세스
1. 큰 기능을 **독립적으로 검증 가능한 작은 단위**로 분해한다.
2. 각 세부 기능의 **전제조건·후행 작업**을 명시한다.
3. **우선순위**(필수/선택)와 **Phase**(Phase 1·2…)를 부여한다.
4. **의존성 그래프**를 표시한다 (Mermaid 가능).

## 출력
`outputs/<오늘날짜>/requirements.md` 에 저장:

```markdown
## 요구사항 분해: "<기능명>"

### Phase 1 (필수)
- [ ] [REQ-1] 세부기능 — 전제: …, 후행: …
- [ ] [REQ-2] …

### Phase 2 (선택)
- [ ] [REQ-3] …

### 의존성
\`\`\`mermaid
flowchart LR
  REQ-1 --> REQ-2 --> REQ-3
\`\`\`
```

## ⚠️ 가드레일 — 사람 승인 필수
분해 결과를 저장한 뒤, **승인 요청**을 출력하고 멈춘다. 사용자의 "승인" 없이 다음 스킬로 진행하지 않는다.

```
이 분해가 요구사항을 정확히 반영했는지 확인해주세요:
- [ ] spec.md 와 충돌 없음
- [ ] 모든 요구사항 포함 (누락 없음)
- [ ] 기존 아키텍처와 충돌 없음
"승인" 또는 "수정 요청"을 알려주세요.
```

## 검증
저장 전, spec.md 의 각 관련 FR 이 분해 결과에 매핑되는지 ✓/✗ 표로 자가검증한다.
