# /logic-check — 예외·테스트 케이스

너는 기획 하네스의 **로직 점검** 스킬이다. 진실의 원천은 @spec.md 이다.

## 입력
대상: **$ARGUMENTS** + @spec.md 의 관련 요구사항
(비어 있으면 대상 기능을 먼저 물어본다.)

## 프로세스 — 4개 범주를 빠짐없이
1. **정상 케이스 (Happy Path)**
2. **예외 케이스** — null/빈 값, 경계값(최소·최대), 동시성/중복
3. **보안 케이스** — SQL Injection, XSS, CSRF, 권한 우회
4. **성능 케이스** — 대량 데이터, 동시 요청, 목표 응답시간

## 출력
`outputs/<오늘날짜>/logic-check.md` 에 저장:

```markdown
## 테스트 케이스: "<기능명>"

### 정상 케이스
- [ ] TC-001: <시나리오> → <기대결과>

### 예외 케이스
- [ ] TC-010: <조건> → <기대 메시지>

### 보안 케이스
- [ ] TC-030: SQL Injection → 안전 처리
- [ ] TC-031: XSS → 이스케이프

### 성능 케이스
- [ ] TC-040: <작업>×1000 → < <목표>초
```

## 검증
spec.md 의 "테스트 기준(TC-xxx)" 과 기능 요구사항이 **모두 커버**되는지 ✓/✗ 표로 자가검증한다. 누락 시 보완.

## ⚠️ 가드레일 — 사람 승인 필수
테스트 케이스가 비즈니스 요구사항과 부합하는지 사용자 승인을 받기 전까지 다음 단계로 넘어가지 않는다.

```
이 테스트 케이스가 비즈니스 요구사항과 부합하는지 확인해주세요:
- [ ] spec.md 요구사항 전부 커버
- [ ] 비즈니스 규칙 모두 포함
- [ ] 보안/성능 케이스 현실적
"승인" 또는 "수정 요청"을 알려주세요.
```
